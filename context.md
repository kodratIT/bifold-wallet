Incoming DIDComm basic messages are most likely being dropped from the UI by connection-scoped state selection, not by the final chat renderer.

- Highest-probability root cause: `connectionId` filtering is the choke point. `useBasicMessagesByConnectionId` only returns records where `m.connectionId === connectionId` in `packages/react-hooks/src/BasicMessageProvider.tsx:26-34`. `useChatMessagesByConnection` depends entirely on that filtered list in `packages/core/src/hooks/chat-messages.tsx:80-92`. If inbound records are stored/logged with a missing, different, or not-yet-resolved `connectionId`, they will be present in provider state/logs but never reach `Chat`.
- Evidence that this matches the symptom: the provider logs only `record.id` on add/update/remove in `packages/react-hooks/src/BasicMessageProvider.tsx:59-74`, so logs can prove receipt/subscription firing without proving the record matches the active chat `connectionId`. This is exactly the pattern “received according to logs, absent in UI”.
- Second likely cause: event-flow timing/gap during initial load. `BasicMessageProvider` first does `findAllByQuery({})` and only subscribes after `loading` becomes false in `packages/react-hooks/src/BasicMessageProvider.tsx:47-57`. Any record saved after the query snapshot but before subscriptions are attached will be missed until a later full refresh/remount. That can produce intermittent “agent received it, UI never showed it” behavior.
- Event-flow is otherwise mostly wired correctly: `recordsAddedByType`/`recordsUpdatedByType` subscribe to repository events and filter by record `type` in `packages/react-hooks/src/recordUtils.ts:59-109`; `useChatMessagesByConnection` recomputes whenever `basicMessages` changes in `packages/core/src/hooks/chat-messages.tsx:90-92,260`; `Chat` passes that array directly to `GiftedChat` in `packages/core/src/screens/Chat.tsx:132-150`. So if a matching basic message reaches provider state, it should normally render.
- Rendering bug looks low probability from these files alone. Basic messages are transformed into valid chat items with `_id`, `text`, `createdAt`, `user`, and custom `renderEvent` in `packages/core/src/hooks/chat-messages.tsx:91-135`. `GiftedChat` receives `messages={chatMessages}` and a stable id generator in `packages/core/src/screens/Chat.tsx:132-139`. Nothing here selectively hides inbound basic messages once they are in `chatMessages`.
- The “mark seen” effect is unlikely to be the primary cause. In `packages/core/src/screens/Chat.tsx:69-90`, it only updates metadata (`customMetadata.seen`) and writes back through the repository. `packages/core/src/types/metadata.ts:10-16` confirms only `seen` is stored. This may trigger an update event, but there is no code here that filters unseen/seen messages out of the list.
- Subtle but lower-confidence event issue: `updateRecord` in `packages/react-hooks/src/recordUtils.ts:36-46` only replaces an existing record and does nothing if the updated record id is not already in state. If a message is first observed by the app as `RecordUpdated` instead of `RecordSaved`, it would never be inserted.
